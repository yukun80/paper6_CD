# 工作日志（2026-03-12）

## 1. 本次目标
- 排查 `sar_prompt_flood.train_reference_prompt` 首轮训练出现 `train_loss=NaN, val_iou=0.0` 的原因。
- 结合参考训练集原始像素值分布，确认当前 `pre/post` 的物理含义是否已经是 dB/log 值。
- 将 `sar_prompt_flood` 特征构造从“逐 tile 相对归一化”调整为“固定 dB 标尺 + 物理一致差分”。

## 2. 关键排查结论

### 2.1 当前像素值按 dB/log 处理更合理
- 抽样统计 `datasets/urban_sar_floods_test/urban_sar_floods_test_tiles_512_band5_band7/pre|post/*.tif`：
  - 常见值域约 `[-50, 7]`
  - 全局最小值约 `-60.0`
  - 中位数约 `-13.5`
  - 平均负值像素比例约 `99.86%`
- 该分布与 SAR backscatter 的 dB/log 表示一致，不像线性幅度域。
- 结论：
  - 当前 `pre/post` 不应再视为线性幅度
  - 对其再次构造“对数比”并不物理一致

### 2.2 `loss=NaN` 的根因不是损失函数，而是特征图先被污染
- 通过逐 batch 检查发现：
  - 某些 query 样本在进入网络前 `features` 已经包含 `NaN`
  - 进入 `BatchNorm` 后进一步污染 running stats，整轮训练随即失稳
- 已定位的具体坏样本：
  - `20210727_Weihui_SAR_r04096_c17408`
- 污染链路：
  - `local_change_strength()` 直接对含无效值的数组做 blur
  - `local_contrast_score -> change_score -> stable_score -> boundary_score` 一起带 `NaN`
  - 最终 `model_input/features` 带 `NaN`

## 3. 代码修改

### 3.1 NaN 稳定性修复
- 修改 `sar_prompt_flood/feature_utils.py`
  - 所有分位数统计仅对 finite 值计算
  - `local_change_strength()` 改为显式接收 `valid_mask`
  - 对无效像素先做安全填充，再做局部均值/方差估计
  - 输出统一做 finite 清洗
- 修改 `sar_prompt_flood/reference_data.py`
  - `build_reference_feature_maps()` 对每个中间图统一清洗 `NaN/Inf`
  - 若关键通道仍残留非有限值，抛出带 `sample_id` 的异常
- 修改 `sar_prompt_flood/train_reference_prompt.py`
  - 训练时增加输入、输出、loss 的 finite 检查
  - 坏 batch 直接跳过并记录 `query_ids/ref_ids`
  - epoch 日志新增：
    - `num_batches`
    - `valid_train_batches`
    - `skipped_batches`

### 3.2 dB 特征设计重构
- 修改 `sar_prompt_flood/reference_data.py`
  - 不再以逐 tile `joint_robust_norm + safe_log_ratio` 为主特征
  - 改为固定 dB 裁剪：
    - `pre/post`：默认裁剪到 `[-35, 5] dB`
    - `delta_db = pre_db - post_db`
    - `darkening_db = relu(delta_db)`
    - `brightening_db = relu(-delta_db)`
  - `diff` 改为裁剪后的 dB 差分归一到 `[-1, 1]`
  - `darkening_score / brightening_score` 改为基于固定 `delta_clip=15 dB` 的物理量缩放
  - `log_ratio_like` 保留接口名，但语义改为 `|delta_db|` 的幅度分数，避免“对 dB 再取 log”
  - `local_contrast_score` 改为在 dB 域直接计算局部统计变化
  - `change_score` 重新改为：
    - `0.60 * darkening_score + 0.25 * local_contrast_score + 0.15 * |delta_db|`
  - `pseudo_rgb` 当前为：
    - `R = pre_db_clip_unit`
    - `G = post_db_clip_unit`
    - `B = darkening_score`
- 修改 `sar_prompt_flood/feature_utils.py`
  - 新增 `clip_unit()`，用于固定物理范围裁剪后的缩放
- 修改配置：
  - `sar_prompt_flood/reference_config.py`
  - `sar_prompt_flood/config/urban_sar_reference.json`
  - 当前 `data_features.channel_set = db_delta_v1`
  - 当前 `data_features.use_joint_norm = false`

## 4. 当前特征语义
- `pre` / `post`：
  - 固定 dB 裁剪后的统一标尺值，不再表示逐景归一化结果
- `diff`：
  - `pre_db - post_db` 的裁剪版，范围约 `[-1, 1]`
- `darkening_score`：
  - 灾后变暗的正向 dB 差，越大越可能是洪水变化
- `brightening_score`：
  - 灾后变亮的正向 dB 差
- `log_ratio_like`：
  - 当前仅保留历史接口名，实际表示 `|delta_db|` 的幅度分数
- `local_contrast_score`：
  - dB 域局部均值/方差变化

## 6. 当前结论
- 当前数据应按 dB/log 域处理，而不是线性幅度域。
- 对 dB 数据做逐 tile 相对归一化，会削弱绝对衰减的物理意义。
- 当前 `sar_prompt_flood` 的特征构造已切换为“固定 dB 标尺 + 差分/局部统计”版本，更符合 SAR 洪水变化的物理解释。
- `loss=NaN` 的直接稳定性问题已修复；下一步应重新跑 `train_reference_prompt`，观察：
  - `train_loss` 是否稳定下降
  - `val_iou` 是否相较旧版逐景归一化特征有改善
